2. TECH STACK
=============

  - Framework:    FastAPI + Uvicorn (async Python)
  - LLM:         Groq Llama-3.3-70B (primary), Vertex AI / Gemini / OpenAI (fallback)
  - Embeddings:  HuggingFace all-MiniLM-L6-v2 (local, zero-cost)
  - Vector DB:   Qdrant (local file-based or cloud)
  - Database:    PostgreSQL (Neon Cloud) / SQLite (dev)
  - Caching:     Redis (optional, for distributed state)
  - Frontend:    Single-file SPA (admin.html) with Tailwind CSS + Vanilla JS
  - Channels:    Web Portal, WhatsApp (Meta Cloud API), Email (Mailgun/SendGrid/Gmail)

4. CUSTOMER MESSAGE FLOW (DETAILED)
====================================

4.1 Entry Points
-----------------

  A) Web Portal:   POST /api/portal/chat
     - Rate limited: 10 requests/minute per IP
     - Security: user_id bound to IP (IDOR prevention)
     - Accepts: form-data with message, user_id, language, file (optional)
     - Handler: portal_routes.py → portal_chat()

  B) WhatsApp:     POST /webhook/whatsapp/
     - Meta Cloud API webhook
     - HMAC signature verification (X-Hub-Signature-256)
     - Handler: whatsapp.py → WhatsAppWebhookService.normalize_payload()
     - Extracts: sender phone, message text, attachments

  C) Email:        POST /webhook/email/
     - Supports Mailgun & SendGrid payload formats
     - Handler: email_handler.py → email_webhook()
     - Extracts: sender, subject, body text

  All three channels converge into:
     → ChatService.process_portal_message(query, user_id, file, language)


4.2 Onboarding State Machine
------------------------------

  New customers go through a multi-step registration flow before AI chat.
  States are persisted in the database (User.state field).

  State Transitions:
  ┌─────┐   greeting    ┌──────────────────┐   detect lang   ┌─────────────┐
  │ new │ ─────────────► │ asking_language   │ ──────────────► │ asking_name │
  └─────┘                └──────────────────┘                 └──────┬──────┘
                                                                     │ name given
                                                                     ▼
                         ┌──────────────────┐   all 4 fields  ┌────────────────┐
                         │confirming_details │ ◄────────────── │ asking_details │
                         └────────┬─────────┘                 └────────────────┘
                                  │ user says "Yes"
                                  ▼
                         ┌──────────────────┐
                         │    complete       │ ──► AI Chat begins
                         └──────────────────┘

  Details collected during onboarding:
    - Language preference (Bahasa Indonesia / English / Chinese)
    - Full name
    - Company name
    - Outlet name
    - Mobile/WhatsApp number
    - Email address

  Supported languages: Indonesian (id), English (en), Chinese (zh)
  All prompts are localized in ChatService.LANG_STRINGS dict.

  EXAMPLE CONVERSATION (new customer "Budi"):

  ──────┼─────────────────────────────────────────┼────────────────────┼─────────────────────────────────────

4.3 AI Processing Pipeline (after onboarding)
-----------------------------------------------

  When a completed user sends a message, this is the exact processing chain:

  STEP 1: Input Guardrail
  ────────────────────────
    File: guardrail_service.py
    - Validates input against prompt injection patterns
    - Rejects malicious/manipulative prompts
    - If rejected → returns safe error message immediately

  STEP 2: Message Saved to DB
  ────────────────────────────
    File: database.py → save_message()
    - Saves user message with role="user" and optional attachments
    - Used for conversation history and transcript generation

  STEP 3: Category/Intent Detection
  ──────────────────────────────────
    File: chat_service.py → _detect_category()
    - Keyword-based multi-category routing
    - Categories and their trigger keywords:

      Category          │ Keywords
      ──────────────────┼──────────────────────────────────────────────
      printer           │ printer, print, receipt, cetak, struk, kds, kiosk
      payment           │ payment, transaction, bayar, refund, void, nets, edc
      voucher           │ voucher, kupon, promo, discount, redeem, loyalty
      diagnose          │ inventory, stock, stok, product, recipe (default)
      pos_guardian       │ "pos-guardian", "pos guardian"
      heart_guardian     │ "heart-guardian", "heart guardian"
      relationship_comms │ "relationship-comms", "comms assistant"
      love_agent         │ "love-agent", "love agent", "relationship coach"

  STEP 4: Persona/Prompt Selection
  ─────────────────────────────────
    File: prompt_service.py → get_system_message(category, user_context)
    - Each category maps to a specialized AI persona
    - BASE_PERSONA: General POS support specialist
    - POS_GUARDIAN_OMEGA: Autonomous incident response AI for POS systems
    - Other personas for specific domains
    - User context (name, company, outlet) injected into prompt

  STEP 5: Language Detection
  ──────────────────────────
    File: chat_service.py → _detect_message_language()
    - Detects language of the CURRENT message (not onboarding preference)
    - Chinese: CJK character detection
    - Bahasa Indonesia: keyword markers (bagaimana, tolong, bisa, tidak, etc.)
    - English: default for Latin-script messages
    - Ensures AI replies in the same language the user is typing

  STEP 6: Context Injection
  ─────────────────────────
    For messages > 3 words and non-special personas:
    - Appends: "(Context: {company} - {outlet})" to the query
    - Helps RAG retrieve company-specific documentation

  STEP 7: Conversation History Retrieval
  ────────────────────────────────────────
    - Fetches last 8 messages from DB for the user
    - Passed to RAG so AI can understand follow-up questions
    - Prevents repetitive greetings in multi-turn conversations

  STEP 8: RAG Query (Retrieval-Augmented Generation)
  ────────────────────────────────────────────────────
    File: rag_service.py → query()
    Sub-steps:
      a) Semantic Cache Check
         - Hash query → check in-memory cache (TTL: 1 hour)
         - If cached → return immediately (saves LLM cost)

      b) Hybrid Search (AdvancedRetriever)
         - Qdrant vector similarity search (semantic)
         - BM25 keyword search (lexical)
         - Reciprocal Rank Fusion to merge results

      c) Cross-Encoder Re-ranking
         - Re-scores retrieved documents for relevance
         - Top-K documents selected as context

      d) Prompt Assembly
         - System prompt (from PromptService)
         - Document context (retrieved chunks)
         - Conversation history (last 8 messages)
         - User query

      e) LLM Completion
         - Sends assembled prompt to Groq Llama-3.3-70B
         - Timeout: 30 seconds
         - Circuit breaker: opens after repeated failures

  STEP 9: Output Guardrail
  ─────────────────────────
    File: guardrail_service.py → validate_output()
    - PII masking (removes leaked personal data)
    - Hallucination detection
    - Sanitizes internal references for public portal users
    - Strips source file names and internal annotations

  STEP 10: Response Saved + Returned
  ───────────────────────────────────
    - AI response saved to DB with role="bot"
    - Response includes: answer, confidence score, sources, persona used
    - For portal users: sources are stripped (privacy)

  STEP 11: Self-Learning Logger
  ──────────────────────────────
    File: chat_service.py → _log_ai_interaction()
    - Logs to AIInteraction table:
      * query (PII-scrubbed)
      * response (PII-scrubbed)
      * retrieved documents
      * tools used
      * confidence score
      * resolution status (pending → reviewed by admin)
    - Used for knowledge gap analysis and continuous improvement


4.4 Escalation Flow
---------------------

  Triggered when:
  - AI confidence is below threshold
  - User explicitly requests human help
  - Critical/urgent issue detected

  File: escalation_service.py → escalate()

  Flow:
    1. Determine severity:
       - is_critical=True  → prefix "A-" → status ON_HOLD (Urgent)
       - is_critical=False → prefix "B-" → status PENDING_PROGRAMMER (High)

    2. Create ticket via TicketService:
       - Summary: "[A-/B-] Escalation: {reason}"
       - Full conversation transcript attached
       - Priority: "Urgent" or "High"

    3. Auto-assignment:
       - ticket_repo.assign_to_least_loaded_agent()
       - Picks the agent with fewest open tickets (load balancing)

    4. Response to customer:
       - "I've created support ticket #{id}. Our agent will handle it shortly."


5. TICKET STATE MACHINE
=========================

  File: state_machine.py

  States:
    ON_HOLD            (A-) — Critical issues requiring immediate attention
    PENDING_PROGRAMMER (B-) — Escalated to developer/specialist
    PENDING_USER       (C-) — Waiting for customer response
    PENDING_AGENT      (D-) — Waiting for agent investigation
    CLOSED             (E-) — Resolved

  Valid Transitions:
    ON_HOLD            → PENDING_PROGRAMMER, PENDING_AGENT, CLOSED
    PENDING_PROGRAMMER → PENDING_AGENT, CLOSED
    PENDING_USER       → PENDING_AGENT, ON_HOLD, CLOSED
    PENDING_AGENT      → PENDING_USER, PENDING_PROGRAMMER, ON_HOLD, CLOSED
    CLOSED             → PENDING_AGENT (reopen)

  Status is determined by ticket summary prefix:
    "A-" → ON_HOLD
    "B-" → PENDING_PROGRAMMER
    "C-" → PENDING_USER
    "D-" → PENDING_AGENT
    "E-" → CLOSED

7. WEBSOCKET REAL-TIME COMMUNICATION
======================================

  Three WebSocket endpoints:

  A) /ws/chat/{session_id}
     - Internal agent ↔ customer chat
     - IDOR protection: verifies participant belongs to session
     - Cross-tenant isolation check
     - Messages saved to DB + broadcast to all session participants

  B) /ws/portal/{user_id}
     - Portal customer push channel
     - Origin validation + user_id format check
     - Max 5 connections per user (flood prevention)
     - Heartbeat every 30 seconds
     - Input guardrail on incoming messages

  C) /ws/portal/admin/{user_id}
     - Admin watcher for portal conversations
     - Requires agent authentication (token in query params)
     - Tenant isolation enforced
     - Agents can send messages directly to portal users


8. CHAT CLOSE FLOW
====================

  File: chat_service.py → close_chat()

  Options:
    "close"              — Resolved. Clear messages, no ticket.
    "ticket"             — Create ticket with full transcript, then clear.
    "ticket_and_notify"  — Create ticket + send notification.

  Steps:
    1. Fetch all portal message history
    2. Fetch WhatsApp history (if user_id is a phone number)
    3. Combine into transcript
    4. If option includes "ticket":
       a. Create ticket with transcript
       b. Auto-assign to agent
       c. Link WhatsApp messages to ticket (if applicable)
    5. Clear portal messages from DB
    6. Email transcript to customer (if email configured)
    7. Reset user state to 'idle'

========================

  File: adapters/email_handler.py

  Inbound:
    Mailgun/SendGrid webhook → POST /webhook/email/
    → Extracts: sender, subject, body (from JSON or form-data)
    → Creates virtual user_id: "email_{sender_sanitized}"

11. SECURITY LAYERS
=====================

  Layer 1: CORS Middleware
    - Origin whitelist via ALLOWED_ORIGINS env var
    - Blocks cross-origin requests from unauthorized domains

  Layer 2: CSRF Protection (CustomCSRFMiddleware)
    - Requires X-CSRF-Token or X-Requested-With header
    - Only for cookie-authenticated state-changing requests (POST/PUT/DELETE)
    - Exempt: /api/portal/, /webhook/, /api/auth/, /api/plans

  Layer 3: Rate Limiting (slowapi)
    - Per IP address
    - Portal chat: 10/min, KB query: 20/min, Registration: 10/min

  Layer 4: Input Guardrail
    - Prompt injection detection
    - Rejects manipulative/malicious prompts before reaching LLM

  Layer 5: Output Guardrail
    - PII masking in AI responses
    - Hallucination detection
    - Source file name stripping for public users

  Layer 6: Authentication (JWT)
    - Access tokens: 60-minute expiry (httponly cookies)
    - Refresh tokens: 7-day expiry
    - MFA: 6-digit code, 5-minute expiry, max 5 attempts
    - Magic link login supported

  Layer 7: Tenant Isolation (TenantMiddleware)
    - Multi-tenant DB scoping via tenant_id
    - Cross-tenant access blocked at query level

  Layer 8: IDOR Prevention
    - Portal user_id bound to IP address
    - WebSocket session ownership verified
    - Ticket access scoped by tenant

  Layer 9: AI Concurrency Gate
    - Semaphore: max 10 simultaneous LLM calls
    - Prevents Groq API quota exhaustion

  Layer 10: Security Headers (SecurityHeadersMiddleware)
    - HSTS (Strict-Transport-Security)
    - X-Content-Type-Options: nosniff
    - X-Frame-Options (clickjacking prevention)

  Layer 11: Correlation ID (CorrelationIDMiddleware)
    - Every request gets a unique X-Trace-ID
    - Propagated through logs for end-to-end debugging
